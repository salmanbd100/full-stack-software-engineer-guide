# English Fluency for Technical Interviews

Build the English fluency needed for senior-level technical interviews at international companies.

## Grammar: Most Common Interview Mistakes

### Present Perfect vs Past Simple

This is the most common mistake non-native speakers make in interviews.

**Present Perfect** (I've worked...) = still relevant now, ongoing, or unspecified time
**Past Simple** (I worked...) = completed, specific past time

| Situation | Correct Form | Example |
|-----------|-------------|---------|
| Skill you still use | Present Perfect | "I've worked with React for 5 years." |
| Role that ended | Past Simple | "I worked at Company X from 2020 to 2023." |
| STAR story actions | Past Simple | "I identified the bottleneck. I implemented code splitting." |
| Experience up to now | Present Perfect | "I've led 10+ projects." |

**❌ Common mistake:**
"I worked with TypeScript for 3 years" (sounds like you stopped)

**✅ Correct:**
"I've worked with TypeScript for 3 years" (still using it)

### Conditionals for Trade-off Discussions

Use these when comparing approaches or discussing alternatives.

**Second Conditional** (Hypothetical present/future):
> "If we used GraphQL, we would reduce network requests by 60%."
> "If I had more time, I would add more test coverage."

**Third Conditional** (Reflecting on past decisions):
> "If we had added testing earlier, we would have caught that bug."
> "If I had known about this pattern, I would have used it from the start."

**Interview application:**
```
"If we chose Approach A, we would get better performance, but if we
chose Approach B, we would have simpler code and faster delivery."
```

### Modal Verbs

| Situation | Modal | Example |
|-----------|-------|---------|
| Demonstrating skill | can | "I can build accessible UIs with WCAG compliance." |
| Making recommendations | should | "We should add error handling for edge cases." |
| Discussing alternatives | would | "I would use Next.js for this use case." |
| Polite requests | could | "Could you repeat the question?" |

## Speaking Fluency

### Reduce Filler Words

Common fillers: *um, uh, like, you know, basically, so, well, actually*

**Strategy:**
1. Record yourself — count fillers per minute
2. Replace fillers with a brief silent pause (2–3 seconds is natural)
3. Use transition phrases instead:
   - "Let me think about that..."
   - "That's a good point. From my experience..."
   - "Give me a moment to organize my thoughts..."

**❌ Before:**
"So, um, I think, like, the best approach is, uh, to use React hooks because,
well, they're more, like, modern and they reduce boilerplate."

**✅ After:**
"I think React hooks are the best approach here. They reduce boilerplate and
make logic reuse much cleaner than class-based alternatives."

---

### Speaking Speed

| Rate | Assessment |
|------|------------|
| < 100 wpm | Too slow — loses listener |
| 130–160 wpm | Optimal — conversational, clear |
| > 180 wpm | Too fast — hard to follow |

**In interviews:** Aim for the lower end (~130 wpm). Non-native speakers
tend to rush when nervous. Slow down deliberately.

## Essential Technical Vocabulary

### Frontend Core

| Term | Example Sentence |
|------|-----------------|
| Reconciliation | "React's reconciliation algorithm updates only the changed DOM nodes." |
| Hydration | "Hydration attaches event listeners to server-rendered HTML." |
| Memoization | "I used useMemo to memoize expensive calculations." |
| Code splitting | "Code splitting reduced initial bundle from 800KB to 250KB." |
| Tree shaking | "Tree shaking eliminated unused exports, cutting bundle size by 30%." |
| Debouncing | "Debouncing limits the search API to one call per 300ms pause." |

### Performance Metrics

| Acronym | Full Name | Use In Sentence |
|---------|-----------|-----------------|
| LCP | Largest Contentful Paint | "LCP improved from 3.2s to 1.5s." |
| CLS | Cumulative Layout Shift | "We fixed CLS by reserving space for lazy-loaded images." |
| TTI | Time to Interactive | "TTI dropped from 4s to 1.6s after code splitting." |
| TTFB | Time to First Byte | "High TTFB for APAC users suggests CDN issues." |

### Algorithms & Complexity

| Notation | Say This |
|----------|----------|
| O(1) | "Constant time" |
| O(log n) | "Logarithmic time" |
| O(n) | "Linear time" |
| O(n²) | "Quadratic time" |

Practice saying: *"This runs in linear time because we traverse the array once.
The hash map provides constant-time lookup."*

### System Design Terms

Practice sentences:
- "We use a CDN to reduce latency for users in different regions."
- "The load balancer distributes traffic across service instances."
- "Database sharding partitions data to improve write throughput."
- "A circuit breaker prevents cascading failures when a service goes down."
- "Eventual consistency is acceptable here — we don't need strict ordering."

## Professional Phrases

### Handling Questions You Don't Know

```
"I'm not deeply familiar with [X], but based on what I know, I'd say [reasoning].
I'd want to confirm that by [checking docs / testing / researching]."
```

### Buying Time to Think

```
"That's a good question. Let me think about that for a moment..."
"Let me approach this systematically..."
"Give me a second to organize my thoughts..."
```

### Asking for Clarification

```
"Just to make sure I understand — you're asking about [paraphrase]?"
"Could you elaborate on [specific part]?"
"When you say [term], do you mean [interpretation A] or [interpretation B]?"
```

### Correcting Yourself

```
"Actually, let me rephrase that..."
"To be more precise..."
"I should clarify — what I meant was..."
```

## Quick Daily Practice

**15 minutes/day covers the 80%:**

1. **5 min** — Record yourself explaining one technical concept from memory
2. **5 min** — Listen back and note fillers or grammar errors
3. **5 min** — Re-record with corrections applied

**Topics to rotate:**
- "Explain how React hooks work"
- "What is the event loop?"
- "How does code splitting improve performance?"
- "Describe a trade-off you made in a recent project"

> **Key insight:** Fluency comes from consistent speaking practice — not grammar study. Record yourself daily.
