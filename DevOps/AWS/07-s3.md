# S3 (Simple Storage Service)

Object storage for any amount of data. Not a filesystem — organized by bucket and key, with no real folder hierarchy.

## Core Concepts

| Term | Definition |
|------|-----------|
| **Bucket** | Container for objects. Name must be globally unique across all AWS accounts. |
| **Object** | A file plus its metadata. Max size: 5 TB. |
| **Key** | The object's unique identifier within a bucket. Looks like a path (`images/avatar.png`) but is actually a flat key. |
| **Region** | Bucket lives in one region. Data does not leave that region unless you replicate it. |

```
s3://my-company-assets/images/avatar.png
       └─ bucket ──┘  └──── key ────────┘
```

## Storage Classes

Choose based on how often you access data and how fast you need it.

| Storage Class | Access | Availability | Cost | Best For |
|--------------|--------|-------------|------|---------|
| **Standard** | Instant | 99.99% | Highest | Frequently accessed data |
| **Intelligent-Tiering** | Instant | 99.9% | Auto-optimized | Unknown or changing access patterns |
| **Standard-IA** | Instant | 99.9% | Lower + retrieval fee | Infrequent access, kept 30+ days |
| **One Zone-IA** | Instant | 99.5% | 20% cheaper than IA | Re-creatable infrequent data |
| **Glacier Instant** | Instant | 99.9% | Very low | Archives needing ms retrieval |
| **Glacier Flexible** | Minutes to hours | 99.99% | Lower | Long-term archives, flexible retrieval |
| **Glacier Deep Archive** | 12 hours | 99.99% | Lowest | Data retained for 7–10 years |

> ✅ Use **Intelligent-Tiering** when you're unsure about access patterns. It moves objects automatically — no retrieval fees for frequent/infrequent tiers.

## Versioning

Versioning keeps every version of an object. Deleting an object adds a **delete marker** instead of removing it.

```bash
# Enable versioning on a bucket
aws s3api put-bucket-versioning \
  --bucket my-company-assets \
  --versioning-configuration Status=Enabled

# List all versions of an object
aws s3api list-object-versions \
  --bucket my-company-assets \
  --prefix images/avatar.png
```

✅ Enable versioning to protect against accidental deletes and overwrites.
⚠️ Versioning increases storage costs. Pair with lifecycle policies to expire old versions.

## Lifecycle Policies

Automate transitions between storage classes and delete old objects.

```json
{
  "Rules": [
    {
      "ID": "archive-and-expire",
      "Status": "Enabled",
      "Filter": { "Prefix": "logs/" },
      "Transitions": [
        { "Days": 30, "StorageClass": "STANDARD_IA" },
        { "Days": 90, "StorageClass": "GLACIER_INSTANT_RETRIEVAL" }
      ],
      "Expiration": { "Days": 365 }
    }
  ]
}
```

```bash
# Apply the lifecycle policy
aws s3api put-bucket-lifecycle-configuration \
  --bucket my-company-assets \
  --lifecycle-configuration file://lifecycle.json
```

## Access Control

This is the most interview-critical section. Know the difference between each mechanism.

### IAM Policies vs Bucket Policies

| | IAM Policy | Bucket Policy |
|--|-----------|--------------|
| **Attached to** | IAM user, role, or group | S3 bucket (resource) |
| **Controls** | What the principal can do | Who can access the bucket |
| **Cross-account** | ❌ Cannot grant cross-account alone | ✅ Can grant access to other AWS accounts |
| **Anonymous access** | ❌ No | ✅ Yes (if public access block is off) |

**Example bucket policy — allow another account to read:**

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "CrossAccountRead",
      "Effect": "Allow",
      "Principal": {
        "AWS": "arn:aws:iam::987654321098:root"
      },
      "Action": ["s3:GetObject"],
      "Resource": "arn:aws:s3:::my-company-assets/*"
    }
  ]
}
```

```bash
# Apply bucket policy
aws s3api put-bucket-policy \
  --bucket my-company-assets \
  --policy file://bucket-policy.json
```

### Public Access Block

Four settings that block public access at the account or bucket level.

| Setting | What It Blocks |
|---------|---------------|
| `BlockPublicAcls` | Rejects requests that grant public ACLs |
| `IgnorePublicAcls` | Ignores any existing public ACLs |
| `BlockPublicPolicy` | Rejects bucket policies that grant public access |
| `RestrictPublicBuckets` | Restricts access to only authorized principals |

```bash
# Block all public access (recommended default)
aws s3api put-public-access-block \
  --bucket my-company-assets \
  --public-access-block-configuration \
    BlockPublicAcls=true,IgnorePublicAcls=true,\
    BlockPublicPolicy=true,RestrictPublicBuckets=true
```

✅ Enable all four settings by default.
❌ Don't disable public access block unless you have a specific need (like a public static site).

> ⚠️ Even with a permissive bucket policy, public access block overrides it.

### ACLs (Legacy)

Access Control Lists are the old way to grant permissions on individual objects. AWS recommends disabling ACLs and using bucket policies instead.

❌ Don't use ACLs for new buckets — bucket policies are more powerful and easier to audit.

## Encryption

| Type | Key Management | Use Case |
|------|---------------|---------|
| **SSE-S3** | AWS manages everything | Default, zero-overhead |
| **SSE-KMS** | You manage keys via KMS | Audit trail, key rotation, compliance |
| **SSE-C** | You provide key per request | You manage keys outside AWS |
| **Client-side** | You encrypt before upload | Maximum control |

```bash
# Upload with SSE-KMS encryption
aws s3 cp file.csv s3://my-company-assets/ \
  --sse aws:kms \
  --sse-kms-key-id arn:aws:kms:us-east-1:123456789012:key/abc-123

# Enforce encryption via bucket policy
# (Deny any PUT without server-side encryption)
```

```json
{
  "Statement": [
    {
      "Effect": "Deny",
      "Principal": "*",
      "Action": "s3:PutObject",
      "Resource": "arn:aws:s3:::my-company-assets/*",
      "Condition": {
        "StringNotEquals": {
          "s3:x-amz-server-side-encryption": "aws:kms"
        }
      }
    }
  ]
}
```

> ✅ Use **SSE-KMS** in regulated environments — it gives you CloudTrail logs for every key usage.

## S3 Event Notifications

Trigger downstream services when objects are created or deleted.

```
S3 Upload Event
    ↓
EventBridge / Lambda / SQS / SNS
    ↓
Process: resize image, parse CSV, index document
```

```bash
# Enable EventBridge for all S3 events (recommended over legacy notifications)
aws s3api put-bucket-notification-configuration \
  --bucket my-company-assets \
  --notification-configuration '{"EventBridgeConfiguration": {}}'
```

> ✅ Use **EventBridge** as the S3 event target — it's more flexible than direct Lambda/SQS triggers.

## Static Website Hosting

Host a single-page app or static site directly from S3.

```bash
# Enable website hosting
aws s3 website s3://my-site/ \
  --index-document index.html \
  --error-document index.html

# Sync local build to S3
aws s3 sync ./dist s3://my-site/ --delete
```

> ⚠️ S3 website endpoints are **HTTP only**. Put **CloudFront** in front for HTTPS and custom domains.

## Large File Uploads

For files over 100 MB, use multipart upload. S3 Transfer Acceleration speeds up uploads to distant regions.

| Feature | When to Use |
|---------|------------|
| **Multipart Upload** | Files > 100 MB. Required for > 5 GB. Parallel parts, resume on failure. |
| **Transfer Acceleration** | Users uploading from far away. Routes through CloudFront edge. |

```bash
# aws s3 cp uses multipart automatically for large files
aws s3 cp large-video.mp4 s3://my-company-assets/ \
  --multipart-threshold 64MB \
  --multipart-chunksize 16MB
```

## Common CLI Commands

```bash
# Create bucket
aws s3 mb s3://my-company-assets --region us-east-1

# Copy file
aws s3 cp ./report.csv s3://my-company-assets/reports/

# Sync directory (like rsync)
aws s3 sync ./dist s3://my-site/ --delete

# List objects
aws s3 ls s3://my-company-assets/ --recursive --human-readable

# Delete object
aws s3 rm s3://my-company-assets/reports/old-report.csv

# Generate presigned URL (expire in 1 hour)
aws s3 presign s3://my-company-assets/private/doc.pdf \
  --expires-in 3600
```

## Interview Q&A

**Q: What is the difference between S3, EBS, and EFS?**

S3 is object storage — great for files, images, backups, and static sites. You access it via HTTP/SDK. It scales infinitely and is cheap. EBS is a block-level disk attached to a single EC2 instance — like a hard drive. Use it for your OS, databases, or anything that needs low-latency disk I/O. EFS is a managed NFS filesystem that multiple EC2 instances can mount at the same time. Use it when you need shared file access across instances, like a CMS or shared config.

---

**Q: How do you make an S3 object publicly accessible?**

Two things must be true. First, the bucket's public access block settings must not block public policies. Disable `BlockPublicPolicy` and `RestrictPublicBuckets`. Second, the bucket policy must explicitly allow `s3:GetObject` to `"Principal": "*"`. Both conditions must be met — a permissive bucket policy alone is blocked by public access block settings.

---

**Q: How does S3 versioning protect against data loss?**

When versioning is enabled, S3 keeps every version of every object. Deleting an object adds a delete marker — the actual data is still there. You can restore it by deleting the marker. Overwriting an object creates a new version — the old version is preserved. This protects against accidental deletes, overwrites, and ransomware (which might overwrite your files). Pair versioning with MFA Delete for extra protection.

---

**Q: What is the difference between an IAM policy and a bucket policy?**

An IAM policy is attached to an IAM identity (user, role, or group) and controls what that identity can do across AWS services. A bucket policy is attached to the S3 bucket itself — it's resource-based. Bucket policies can grant cross-account access (other AWS accounts) and can allow anonymous public access. For cross-account S3 access, you need a bucket policy — IAM alone isn't enough.

---

**Q: When would you use S3 Intelligent-Tiering?**

Use Intelligent-Tiering when you don't know how often objects will be accessed. It monitors access patterns and automatically moves objects between frequent, infrequent, archive, and deep archive tiers. There's a small monitoring fee per object, so it's not cost-effective for millions of tiny objects or for data you know is accessed frequently. For predictable patterns, use explicit lifecycle rules instead.
