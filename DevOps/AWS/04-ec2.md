# EC2 (Elastic Compute Cloud)

EC2 gives you virtual machines on AWS. You control the OS, software, and configuration — AWS manages the physical hardware.

## What EC2 Is

Think of EC2 like renting a computer in a data center. You pick the size, the operating system, and how long you need it. AWS handles the rack, the power, and the network.

This is **Infrastructure as a Service (IaaS)**. You are responsible for everything above the hypervisor: OS patching, app deployment, security groups, and backups.

## Instance Type Families

Pick the right family first. Then pick the size (small → 4xlarge).

| Family | Examples | Optimized For | Typical Use Case |
|--------|----------|---------------|-----------------|
| **General Purpose** | t3, m5, m6i | Balanced CPU/RAM | Web servers, small DBs, dev environments |
| **Compute Optimized** | c5, c6i | High CPU | Batch processing, ML inference, game servers |
| **Memory Optimized** | r5, r6i, x1 | High RAM | In-memory caches, large databases, SAP HANA |
| **Storage Optimized** | i3, i4i, d3 | High IOPS / throughput | NoSQL DBs, data warehousing, Elasticsearch |
| **Accelerated** | p3, g4dn | GPU | ML training, video encoding, HPC |

> **t3** instances use **burstable CPU credits**. Great for dev workloads with variable traffic. Bad for sustained high-CPU tasks — they throttle when credits run out.

**Naming convention:** `c5.2xlarge` = compute family (c), 5th generation, 2× large size.

## AMIs (Amazon Machine Images)

An AMI is a **snapshot of a configured machine**. It captures the OS, installed software, and configuration. You launch an EC2 instance from an AMI.

```
AMI
├── Root volume snapshot (OS + software)
├── Launch permissions (who can use it)
└── Block device mapping (which EBS volumes to attach)
```

**Three sources:**

| Source | Description | Use Case |
|--------|-------------|----------|
| **AWS-provided** | Amazon Linux 2, Ubuntu, Windows | Baseline starting point |
| **Marketplace** | Third-party (e.g. hardened CIS images) | Pre-configured software stacks |
| **Custom AMI** | You build it | Golden image for your fleet |

**Why custom AMIs?** They make instances start faster. Instead of running a 5-minute bootstrap script on every new instance, bake the software into the AMI. The instance is ready in seconds.

✅ Use custom AMIs for production Auto Scaling Groups — fast, consistent, auditable.

❌ Don't rely on User Data alone for production fleets — it runs on every boot and is slow.

## User Data

User Data is a **bootstrap script** that runs once on first launch. Use it for light configuration or to pull the latest app version.

```bash
#!/bin/bash
# Install Node.js and start the app on first boot
yum update -y
curl -sL https://rpm.nodesource.com/setup_18.x | bash -
yum install -y nodejs
cd /opt/app
npm install
node server.js &
```

⚠️ User Data runs as **root** on first launch only. It does not re-run on stop/start.

## Instance Metadata

Every EC2 instance can query its own metadata from a special local endpoint. No credentials needed — only accessible from within the instance.

```bash
# Get the instance ID
curl http://169.254.169.254/latest/meta-data/instance-id

# Get the current IAM role credentials
curl http://169.254.169.254/latest/meta-data/iam/security-credentials/MyRole

# Get the Availability Zone
curl http://169.254.169.254/latest/meta-data/placement/availability-zone
```

**Why it matters:** This is how apps running on EC2 get temporary credentials, the instance ID for logging, and the AZ for routing decisions — without hardcoding anything.

⚠️ IMDSv2 is the secure version. It requires a session token first. Enable it to prevent SSRF attacks that steal credentials via the metadata endpoint.

```bash
# IMDSv2: get a token first, then use it
TOKEN=$(curl -X PUT "http://169.254.169.254/latest/api/token" \
  -H "X-aws-ec2-metadata-token-ttl-seconds: 21600")

curl -H "X-aws-ec2-metadata-token: $TOKEN" \
  http://169.254.169.254/latest/meta-data/instance-id
```

## Auto Scaling Groups (ASG)

ASG is the **most important EC2 topic for DevOps interviews**. It automatically adds or removes instances based on demand.

```
Auto Scaling Group
├── Launch Template       — what to launch (AMI, type, SG, IAM role)
├── Min / Desired / Max   — capacity bounds
├── Scaling Policies      — when to scale
└── Health Checks         — when to replace unhealthy instances
```

### Launch Template vs Launch Configuration

| | Launch Template | Launch Configuration |
|-|----------------|---------------------|
| **Status** | ✅ Current | ❌ Legacy (no new features) |
| **Versioning** | Yes — multiple versions | No |
| **Spot support** | Full (mixed instances) | Limited |
| **T2/T3 unlimited** | Yes | No |

✅ Always use **Launch Templates**. AWS no longer adds features to Launch Configurations.

### Capacity Settings

```
Min = 2   ← ASG will never go below this (protects against full outage)
Desired = 4   ← ASG tries to keep this many running
Max = 10  ← ASG will never exceed this (cost control)
```

### Scaling Policies

| Policy | How It Works | Best For |
|--------|-------------|----------|
| **Target Tracking** | Keeps a metric at a target (e.g. CPU = 50%) | Most use cases — simple and automatic |
| **Step Scaling** | Adds/removes instances in steps based on alarm thresholds | Fine-grained control |
| **Scheduled** | Scales at a fixed time | Predictable traffic (e.g. market open) |

✅ **Target tracking on CPU 50%** is the default starting point for most web workloads.

### Health Checks

| Type | What It Checks | When to Use |
|------|----------------|-------------|
| **EC2 health check** | Instance status (running, stopped) | Default — always on |
| **ELB health check** | App responds to HTTP ping | ✅ Use this for web apps |

⚠️ EC2 health check only replaces instances that are completely stopped or terminated. If your app crashes but the instance keeps running, EC2 health check won't catch it. Enable **ELB health checks** so ASG replaces instances with failed apps.

## Pricing Models

| Model | Cost | Commitment | Best For |
|-------|------|------------|----------|
| **On-Demand** | Highest | None | Dev, unpredictable workloads |
| **Reserved (1-year)** | ~40% off | 1 year | Steady-state production |
| **Reserved (3-year)** | ~60% off | 3 years | Long-term baseline |
| **Savings Plans** | ~40-60% off | 1–3 years, flexible | Flexible compute commitment |
| **Spot** | Up to 90% off | None | Fault-tolerant batch jobs |

**Spot Instances** are spare AWS capacity sold cheaply. AWS can reclaim them with a **2-minute warning**. Design for interruption: use Spot for stateless workers, batch jobs, and CI runners — never for databases or session-based apps.

> The winning cost strategy: **Reserved for baseline** + **Spot for burst** + **On-Demand for the gap**.

## Key CLI Commands

```bash
# Launch an instance
aws ec2 run-instances \
  --image-id ami-0abcdef1234567890 \
  --instance-type t3.micro \
  --key-name my-key \
  --security-group-ids sg-12345678 \
  --subnet-id subnet-12345678

# List running instances
aws ec2 describe-instances \
  --filters "Name=instance-state-name,Values=running" \
  --query "Reservations[*].Instances[*].[InstanceId,PublicIpAddress]" \
  --output table

# Stop an instance (keeps EBS volume, keeps Elastic IP reserved)
aws ec2 stop-instances --instance-ids i-1234567890abcdef0

# Terminate an instance (deletes root EBS, releases IP — permanent)
aws ec2 terminate-instances --instance-ids i-1234567890abcdef0
```

## Important Distinctions

| Pair | Difference |
|------|-----------|
| **Stop vs Terminate** | Stop = off (data preserved, restartable). Terminate = deleted (root volume gone by default). |
| **AMI vs Snapshot** | Snapshot = point-in-time copy of an EBS volume. AMI = complete machine image (references one or more snapshots + launch config). You create an AMI from an instance; it creates snapshots behind the scenes. |
| **ASG vs Manual scaling** | ASG replaces unhealthy instances and scales automatically. Manual scaling is just changing instance count — no automation, no self-healing. |
| **Security Group vs NACL** | Security Group = stateful, attached to instances/ENIs. NACL = stateless, attached to subnets. SGs are the primary tool. |

## Interview Q&A

**Q: When would you use Spot instances and when would you avoid them?**

Use Spot for stateless, fault-tolerant workloads: batch data processing, CI/CD runners, ML training jobs. Spot saves up to 90% but can be reclaimed with 2 minutes notice. Avoid Spot for databases, session stores, or any workload that cannot safely handle sudden termination.

**Q: What is an AMI and why would you create a custom one?**

An AMI is a machine image — it contains the OS, software, and configuration needed to launch an EC2 instance. You create a custom AMI (a "golden image") so every instance in your fleet starts from a known, tested state. It removes bootstrap time, reduces configuration drift, and makes rollbacks simple — just launch from the previous AMI version.

**Q: How does an Auto Scaling Group decide to replace an instance?**

ASG uses health checks. If you enable EC2 health checks only, ASG replaces instances that are stopped or in a bad hardware state. If you also enable ELB health checks, ASG replaces instances where the app itself is unhealthy — for example, returning 503. For web services, always enable ELB health checks.

**Q: What is the difference between stop and terminate?**

Stop pauses the instance. The EBS root volume is preserved, the Elastic IP (if allocated) stays reserved, and you can restart it later. You still pay for EBS storage while stopped. Terminate permanently deletes the instance and, by default, deletes the root EBS volume. It cannot be undone.

**Q: What is the difference between a Reserved Instance and a Savings Plan?**

Reserved Instances lock you into a specific instance type and region (e.g. m5.xlarge in us-east-1). Savings Plans commit you to a dollar-per-hour spend across any compute type — EC2, Fargate, or Lambda. Savings Plans are more flexible and the recommended default. Use Reserved Instances only when you need the maximum discount on a fixed workload.
