# End-to-End (E2E) Testing

## 💡 What Is an E2E Test?

An E2E test drives the **whole system from outside** — usually through HTTP, a browser, or a CLI — exactly like a real user or client would. Database, cache, queue, third-party sandboxes: all real.

> If a unit test mocks dependencies, an integration test wires components, an E2E test runs the *entire* application as one black box.

**Tradeoffs:**

- ✅ Highest confidence — proves the system actually works
- ❌ Slowest tier — seconds per test, minutes per suite
- ❌ Flakiest — network blips, async races, container startup

---

## E2E vs Integration

| Aspect             | Integration                        | E2E                                  |
| ------------------ | ---------------------------------- | ------------------------------------ |
| **Scope**          | One service + its DB               | Full deployment, often multi-service |
| **Auth**           | Bypass / inject token              | Real login flow                      |
| **External calls** | Mocked                             | Sandbox or real                      |
| **Run frequency**  | Every PR                           | Pre-release, nightly                 |
| **Count**          | Hundreds                           | Tens                                 |

---

## What to Cover

Pick the **critical user journeys**, not every code path.

**Cover with E2E:**

- ✅ Sign up → verify email → log in
- ✅ Add item to cart → checkout → payment → order confirmation
- ✅ Upload file → process → download result

**Don't cover with E2E:**

- ❌ Every validation error message (use unit tests)
- ❌ Edge cases in pricing calculations (use unit tests)
- ❌ Every CRUD endpoint variation (use integration tests)

> Rule of thumb: ~5–15 E2E tests for a service. Hundreds of unit tests. Tens of integration tests.

## API-Only E2E

When the product is an API, run the whole app and hit it over HTTP.

```typescript
import request from "supertest";

const baseUrl = process.env.API_URL ?? "http://localhost:3000";

describe("Order flow E2E", () => {
  let token: string;
  let userId: string;
  let orderId: string;

  test("user signs up", async () => {
    const res = await request(baseUrl)
      .post("/auth/signup")
      .send({ email: "e2e@example.com", password: "Strong!Pass1" })
      .expect(201);

    token = res.body.token;
    userId = res.body.user.id;
  });

  test("user creates an order", async () => {
    const res = await request(baseUrl)
      .post("/orders")
      .set("Authorization", `Bearer ${token}`)
      .send({ items: [{ sku: "BOOK-1", qty: 2 }] })
      .expect(201);

    orderId = res.body.id;
    expect(res.body.status).toBe("pending");
  });

  test("payment webhook completes the order", async () => {
    await request(baseUrl)
      .post("/webhooks/payment")
      .set("X-Webhook-Signature", validSignature)
      .send({ orderId, status: "paid" })
      .expect(200);

    const order = await request(baseUrl)
      .get(`/orders/${orderId}`)
      .set("Authorization", `Bearer ${token}`);

    expect(order.body.status).toBe("paid");
  });
});
```

## Browser E2E with Playwright

For full-stack apps with a UI, Playwright is the modern default.

```typescript
import { test, expect } from "@playwright/test";

test("checkout flow", async ({ page }) => {
  await page.goto("/");
  await page.getByRole("link", { name: "Sign in" }).click();
  await page.getByLabel("Email").fill("e2e@example.com");
  await page.getByLabel("Password").fill("Strong!Pass1");
  await page.getByRole("button", { name: "Sign in" }).click();

  await page.getByRole("link", { name: "Books" }).click();
  await page.getByRole("button", { name: "Add to cart" }).first().click();
  await page.getByRole("link", { name: "Checkout" }).click();
  await page.getByRole("button", { name: "Pay now" }).click();

  await expect(page.getByText("Order confirmed")).toBeVisible();
});
```

## Fighting Flakiness

E2E flakiness usually comes from **timing**.

**❌ Bad — sleep and pray:**

```typescript
await new Promise((r) => setTimeout(r, 2000));
expect(await getOrder(id)).toBeDefined();
```

**✅ Good — poll with a clear stop condition:**

```typescript
async function waitFor<T>(fn: () => Promise<T | null>, timeout = 5000): Promise<T> {
  const start = Date.now();
  while (Date.now() - start < timeout) {
    const result = await fn();
    if (result) return result;
    await new Promise((r) => setTimeout(r, 100));
  }
  throw new Error("Timeout");
}

const order = await waitFor(() => getOrder(id));
```

**Other anti-flake tactics:**

| Cause                | Fix                                    |
| -------------------- | -------------------------------------- |
| Race on async writes | Poll until expected state              |
| Cross-test pollution | Unique users/emails per test           |
| Time-of-day bugs     | Inject a clock or freeze time          |
| Network blips        | Retry with backoff *for the test only* |

## Environment Setup

E2E tests run against a deployed-style environment:

```yaml
# docker-compose.e2e.yml
services:
  api:
    build: .
    environment:
      DATABASE_URL: postgres://postgres@db/test
  db:
    image: postgres:16
  redis:
    image: redis:7
```

```typescript
// global-setup.ts
import { execSync } from "node:child_process";

export default async function setup() {
  execSync("docker compose -f docker-compose.e2e.yml up -d --wait");
  execSync("npm run migrate");
  execSync("npm run seed:e2e");
}
```

## Third-Party Services

Use the vendor's **sandbox** environment when available:

- Stripe → test mode + test cards
- Twilio → test credentials
- AWS → LocalStack or scoped sub-account

When no sandbox exists, fall back to a recorded mock (Pact, VCR) and run a separate "contract test" against the real service nightly.

## Interview Q&A

**Q: Why not just E2E test everything?**
A: E2E tests are slow, expensive, and flaky. A 500-test E2E suite takes hours and fails on transient network issues. Unit and integration tests catch most bugs cheaper. Reserve E2E for critical user journeys.

**Q: Your E2E suite is flaky — how do you stabilize it?**
A: Replace sleeps with polling, isolate test data (unique IDs per run), pin time and randomness, and audit for shared state. If a single test is flaky, fix the *root cause* — don't add retries to hide it.

**Q: How do you handle E2E tests that need a fresh database?**
A: Spin up a containerized DB per CI run, run migrations + seeds in `globalSetup`, and tear down in `globalTeardown`. For local dev, point tests at a Docker Compose stack.

---

## Best Practices

✅ Cover only critical user journeys
✅ Use real services (or vendor sandboxes) end to end
✅ Poll for async state — never `setTimeout`
✅ Isolate test data with unique IDs
✅ Run nightly in CI; gate releases on green
❌ Don't replicate unit test coverage in E2E
❌ Don't retry flaky tests — fix the cause
❌ Don't share state between tests
